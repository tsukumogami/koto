# Completeness Review

## Verdict: FAIL
The PRD has strong requirements and examples but leaves critical migration mechanics unspecified -- an implementer would have to guess how existing gate types produce structured output and how the engine transitions from boolean to structured evaluation.

## Issues Found

1. **GateResult enum mapping is unaddressed.** The current `GateResult` enum has four variants: `Passed`, `Failed { exit_code }`, `TimedOut`, and `Error { message }`. The PRD requires gates to produce structured data matching an `output_schema`, but never specifies how the engine maps these existing variants to structured output. For example, when a command gate times out or errors, what structured data does it produce? The `output_schema` only covers the happy path (passed/failed). An implementer must decide whether `TimedOut` maps to the failed branch of the schema, produces a separate field, or is handled outside the schema entirely. **Fix:** Add a requirement (or a section in Known Limitations) specifying how `GateResult::TimedOut` and `GateResult::Error` map to structured output, or whether they bypass the schema and remain engine-level stop conditions.

2. **No specification for how each gate type produces structured output.** The PRD says "Each gate type must be updated to produce structured data" (Known Limitations) but the Out of Scope section explicitly defers "how command gates produce structured data from command output." This creates a dependency gap: the PRD requires structured output (R2) but doesn't specify the mechanism for any of the three existing gate types. For `context-exists`, it's straightforward (boolean). For `context-matches`, the gate already has access to matched content but the PRD doesn't define what fields it should expose. For `command`, the PRD punts entirely. **Fix:** Either (a) specify a default output schema per gate type that the engine auto-applies when no `output_schema` is declared, or (b) narrow R2 to only apply when `output_schema` is explicitly provided, and clarify that gates without schemas produce no structured data.

3. **Backward compatibility (R10) conflicts with R1.** R1 says "each gate declares an output_schema." R10 says "existing templates without output_schema continue to compile." These are compatible as stated, but the interaction is underspecified. When a gate has no `output_schema`: Can its transitions use `when` clauses? Do `gates.<name>.*` fields exist in the resolver? How does `resolve_transition` behave when gate output is absent -- does it fall back to the current boolean `gate_failed` flag? **Fix:** Add a requirement explicitly stating what the transition resolver sees when a gate has no output schema. The simplest answer is: gates without schemas behave exactly as today (boolean block/unblock, no `gates.*` fields available to `when` clauses).

4. **No AC for `GateResult::TimedOut` or `GateResult::Error` scenarios.** The acceptance criteria cover passing gates, failing gates, and overrides, but never address what happens when a gate times out or errors. These are distinct from "failed" in the current model. Does a timed-out gate count as "failing" for override purposes? Can you override a gate that errored? **Fix:** Add acceptance criteria: "A timed-out gate is treated as a failing gate for override purposes" and "A gate that errors (e.g., command not found) is treated as a failing gate for override purposes" (or specify different behavior).

5. **No requirement for how `output_schema` is validated against actual gate output at runtime.** R9 covers compile-time validation. But at runtime, what happens if a command gate's JSON output doesn't match its declared `output_schema`? Is this an error? Does the gate count as failed? Is it silently coerced? **Fix:** Add a requirement specifying runtime schema validation behavior, or explicitly defer it to Out of Scope.

6. **`koto overrides list` is introduced without specifying its relationship to existing CLI commands.** The PRD introduces `koto overrides list` (R8, Example 5) as a new CLI command but doesn't specify: Is `overrides` a new top-level subcommand? Does it take a workflow argument or infer from current directory? What's the output format (JSON only, or also human-readable)? **Fix:** Either specify the CLI surface minimally (subcommand name, required arguments, output format) or move CLI specifics to Out of Scope and keep R8 focused on the `derive_overrides` function.

7. **No requirement for what `koto next` output looks like when gates produce structured data.** Example 1 shows `gate_blocked` with `blocking_conditions`, but the current `StopReason::GateBlocked` carries `BTreeMap<String, GateResult>`. The PRD doesn't specify how the `koto next` JSON output changes to include structured gate data. This is the primary agent-facing interface. **Fix:** Add a requirement specifying that `koto next` output includes gate structured output in both the blocked and advanced cases.

8. **Selective override persistence across invocations is unspecified.** Example 4 shows overriding `schema_check` in one call, then `size_check` in a second call. This implies the first override's effect persists. But the current engine re-evaluates gates on every `koto next` call. If the gate command still fails on re-evaluation, does the previous override still apply? Or must the agent override again? **Fix:** Add a requirement specifying whether override state persists within an epoch or whether gates are re-evaluated on each `koto next` and overrides must be resubmitted.

9. **No user story for the "gate passes automatically and routes without agent" scenario.** The PRD's interaction examples show this (Example 1), but no user story captures it. This is arguably the most important behavioral change: gates that pass now auto-route via structured output, eliminating the need for agent interaction on passing deterministic states. **Fix:** Add a user story: "As a template author, I want passing gates to automatically route to the correct transition based on their structured output, so deterministic states don't require agent interaction."

## Suggested Improvements

1. **Add a "Gate type output mapping" section.** A table showing each existing gate type and what structured output it produces would make the design concrete. For example: `context-exists` produces `{exists: bool}`, `context-matches` produces `{matches: bool}` (or `{matches: bool, content: string}`), `command` produces whatever the `output_schema` declares (with a parsing mechanism TBD). This bridges the gap between the abstract requirement and the implementation.

2. **Clarify the relationship between `gate_failed` boolean flag and structured output.** The current `resolve_transition` uses a `gate_failed: bool` parameter. The new model replaces this with structured data in the evidence map. The PRD should state that `gate_failed` is removed (or deprecated) in favor of checking whether `gates.*` fields satisfy `when` clauses. This is a significant change to the transition resolver's algorithm.

3. **Add an interaction example showing backward-compatible behavior.** Show what happens when an existing template (no `output_schema`, no `gates.*` in `when` clauses) is processed by the new engine. This would demonstrate that R10 works as intended.

4. **Consider whether `override_default` should be optional for backward-compatible gates.** R10 says gates without schemas work as today. But R4 says each gate declares an `override_default`. If a gate has no `output_schema`, does it need an `override_default`? The compiler validation in R9 suggests both are required together, but this should be explicit.

5. **Specify event ordering when a gate passes and auto-routes.** R11 covers the override + evidence case. But when a gate passes and produces structured output that feeds into transition routing, is a new event type emitted (e.g., `GateEvaluated`)? Or is the structured output only visible through the transition event? The audit trail for "gate passed with these values, which caused this transition" seems important.

## Summary

The PRD clearly articulates the problem and the target design. The interaction examples are strong and the requirements are well-structured. However, it has a critical gap around the migration from `GateResult` (boolean enum) to structured output -- the mapping mechanism, runtime validation, and persistence of override state across re-evaluations are all unspecified. An implementer would need to make at least 3-4 significant design decisions that the PRD leaves open. Addressing issues 1-4 and 8 would bring it to PASS.
