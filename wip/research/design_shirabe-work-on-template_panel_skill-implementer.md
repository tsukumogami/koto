# Design Review: shirabe work-on koto template
## Panel Role: Skill Implementer

---

## 1. Phase Realism: The 15-state map does not match what the skill actually does

The design names 15 states but the actual /work-on skill has 7 phases, not 15. More importantly, the mapping is structurally wrong in several places.

**Phase 0 (Context Injection) is absent from the state machine.** The real skill runs a shell script (`extract-context.sh`) that creates `wip/IMPLEMENTATION_CONTEXT.md`, which the agent reads and summarizes before starting setup. This is a meaningful step — it's how the agent internalizes design rationale before branching. The design collapses it into `context_injection`, which only checks that the GitHub issue is *accessible*. The gate is `gh issue view {{ISSUE_NUMBER}} --json number --jq .number`. That checks reachability, not design context extraction. An agent following the template's `context_injection` state will call `koto next`, have the gate auto-advance if the issue exists, and move to `setup` without ever reading a design doc. The entire context injection purpose is lost.

**The introspection branch logic is wrong.** In the real skill, Phase 2 runs a staleness check script first. If `introspection_recommended: false`, the phase is skipped entirely and the agent goes directly to analysis. If `introspection_recommended: true`, a sub-agent is spawned via the Task tool to do the actual introspection work and write `wip/issue_<N>_introspection.md`. The design represents this as two states: `staleness_check` and `introspection`. But the design's `staleness_check` evidence enum — `enum[fresh, stale_skip_introspection, stale_requires_introspection]` — conflates the script output with the agent's routing decision. In the real skill, "fresh" is determined by the staleness script's exit code and YAML output; the agent doesn't assess freshness, the script does. The agent only makes a routing decision *after* reading the script output. The design puts the agent in charge of determining freshness with no instruction on how to make that call.

**The introspection state omits the sub-agent spawn.** The real Phase 2 explicitly launches a sub-agent via the Task tool to perform introspection, then handles four possible recommendations: Proceed, Clarify, Amend, Re-plan. Clarify and Amend require AskUserQuestion. Re-plan stops the workflow. The design's `introspection` state has a 3-value evidence enum: `enum[approach_unchanged, approach_updated, issue_superseded]`. These don't map to the actual outcomes. "Clarify" and "Amend" aren't represented. "Re-plan" (which stops the workflow) is missing entirely — `issue_superseded` routes somewhere, but where? The design's state machine diagram doesn't show a `done_blocked` edge from `introspection`. An agent that gets `Re-plan` from the real introspection sub-agent has no state to transition to.

**Implementation's iterative structure is not modeled.** The real Phase 4 is a loop: write code, validate, fix, commit, repeat. The design's `implementation` state models this with a self-loop on `partial_tests_failing`. But the real skill runs the A-B-C-D cycle (Write, Validate, Functional Test, Write Tests) per step, then also spawns optional review sub-agents (security, performance, testing, architecture). These aren't phases in the design. The `implementation` state's directive would need to describe the entire iterative loop, including when to spawn sub-agents. A bare `implementation_status: enum[complete, partial_tests_failing, blocked]` evidence field can't capture "I'm on step 3 of 7 and the tests are failing at step 3."

**Finalization's auto-skip logic is missing.** The real Phase 5 has conditional summary generation: documentation-only issues skip summary creation and go straight to cleanup + final verification. The design's `finalization` state has `finalization_status: enum[ready_for_pr, deferred_items_noted, issues_found]` but no mechanism for routing around the summary creation step based on issue labels. This is a non-trivial branch — in the real skill it's explicit: "Skip summary for issues labeled docs, config, chore." The template would need a template variable or evidence field for issue type to replicate this.

**The design invents a `research` state that doesn't exist in /work-on.** The free-form mode adds a `research` state between `task_validation` and `setup`. There's no equivalent in /work-on. The design drew this from the `/just-do-it` skill's Phase 2 (Research, borrowed from /issue), but the real /work-on doesn't do a separate research step before analysis. Analysis *is* the research. Adding a dedicated research state in the template but not in the skill means the template's directive will instruct the agent to do something the skill doesn't describe, and agents resuming mid-workflow will end up in a state with no guidance from the original skill documentation.

**Count check:** Counting named states from the design: entry, task_validation, validation_exit, research, context_injection, setup, staleness_check, introspection, analysis, implementation, finalization, pr_creation, ci_monitor, done, done_blocked = 15. But the actual /work-on skill maps to roughly: entry, context_injection (if kept), setup, staleness_check, (optional) introspection, analysis, implementation, finalization, pr_creation, ci_monitor, done, done_blocked = 12 states without the free-form additions. The 3 extra states (task_validation, validation_exit, research) are free-form additions. The real mismatch is that the design's state names look right but the state *contents* don't match the actual phase work.

## 2. Directive Quality: The design gives no actual directive text

The design specifies the *structure* of directives — what evidence each state captures, what gates run — but never writes the actual directive text an agent will read. This is the core deliverable of Phase 2 ("Write the template file"), and the design omits it entirely.

**What good directives look like.** The hello-koto template (the only shipped example) writes full prose directives in the template's markdown body, like: "You are <name>, a tsukumogami spirit awakening for the first time. Create a file at `wip/spirit-greeting.txt` containing a greeting from <name> to the world." It's imperative, specific, and contains the exact artifact paths and actions the agent needs.

**What the design gives instead.** The state definitions in the design read like evidence schema documentation, not agent instructions. For `staleness_check`: "Evidence: `staleness_signal: enum[fresh, stale_skip_introspection, stale_requires_introspection]`, `staleness_details: string`." This tells the implementer what fields to capture, not what the agent should actually *do* in this state. The agent calling `koto next` in `staleness_check` will receive the directive from the template — but the design never specifies what that directive says. Should the agent run the staleness check script? How? Where is it? What if it fails? The design's rationale says "Command gates can only check exit codes, not script output content, so the routing decision (fresh/stale) must always be submitted by the agent" — but there's no directive text telling the agent *how* to make that determination.

**The missing instructions are load-bearing.** The real /work-on skill keeps phase instructions in `references/phases/phase-N.md` files. Each file is 50-80 lines of specific procedure. The `staleness_check` state's directive needs to say: run this script, read these fields in the YAML output, if `introspection_recommended: true` set your staleness signal to `stale_requires_introspection`, else set it to `fresh` (or `stale_skip_introspection`). Without those specifics in the directive, agents will either do nothing or hallucinate a procedure.

**The `implementation` state is the worst case.** Phase 4 in the real skill spans 130 lines of instructions covering the A-B-C-D cycle, coverage tracking, handling blockers, self-review, requirements cross-reference, and conditional sub-agent spawning. A koto template directive can't reference external files — it's a string in the template. The design doesn't address how to compress 130 lines of phase instructions into a single directive field, or whether the skill's SKILL.md will supplement the directive. An implementer picking up Phase 2 has no idea what to write in the `implementation` state's directive.

## 3. Evidence Schema Validity: Several enums don't match real agent decisions

**`baseline_outcome: enum[clean, existing_failures, build_broken]`** — This is plausible but incomplete. The real Phase 1 records: test pass/fail counts, any pre-existing failures, build status, AND coverage metrics. The design's enum captures failure type but not whether there were pre-existing failures that should be documented for later comparison. An agent selecting `existing_failures` has no field to record *which* failures — the `staleness_details` analogue is missing. More critically, the routing from `setup` to the next state depends on `mode`, not on `baseline_outcome`. The `baseline_outcome` field in `setup` evidence is purely documentary — it doesn't route anywhere. If it doesn't route, why is it in the evidence schema at all? Evidence fields that don't affect transitions are exactly the "ceremony without enforcement value" the design criticizes in Fine-grained Option.

**`staleness_signal: enum[fresh, stale_skip_introspection, stale_requires_introspection]`** — The real staleness check script (`check-staleness.sh`) outputs a YAML blob with `introspection_recommended: true/false`. The agent's job is to read that output and act on it. The design's enum tries to encode the script's output as an agent decision, but the values aren't aligned. The script returns a binary: introspect or don't. The design adds `stale_skip_introspection` as a distinct state from `fresh`, implying the agent assesses "freshness" independently from whether introspection is recommended. That's not what the real skill does. The agent doesn't independently assess freshness — it reads the script output and routes. A two-value enum (`introspect: true/false`) would be more honest.

**`introspection_outcome: enum[approach_unchanged, approach_updated, issue_superseded]`** — The real Phase 2 sub-agent returns four possible recommendations: Proceed, Clarify, Amend, Re-plan. The design's three-value enum misses Clarify and Amend. More importantly, Clarify and Amend require user interaction (AskUserQuestion), which is not modeled by any state or transition. An agent in `introspection` that gets an "Amend" recommendation from the sub-agent has nowhere to go in the design's state machine.

**`plan_outcome: enum[plan_ready, blocked_missing_context, scope_changed]`** — `scope_changed` routes back to `analysis` (self-loop). In the real skill, if scope changes during analysis the plan file is updated and the agent continues. A self-loop on `scope_changed` is fine. But `blocked_missing_context` routing to `done_blocked` is aggressive — the real Phase 3 says "If blocked during implementation, document the blocker in the plan and ask for clarification if needed." Analysis rarely hard-blocks; it's more likely to surface uncertainty that gets documented in the plan. Routing to `done_blocked` on `blocked_missing_context` means a single ambiguous dependency stops the entire workflow. The real skill doesn't do this.

**`finalization_status: enum[ready_for_pr, deferred_items_noted, issues_found]`** — `issues_found` has no routing in the design's state machine. The diagram shows no edge from `finalization` to any state on `issues_found`. What does the agent do? The design doesn't say. `deferred_items_noted` also lacks routing — it presumably goes to `pr_creation`, but this isn't specified. The real Phase 5 says if tests fail, fix them before proceeding. The design gives no blocking behavior on `issues_found`.

**`ci_outcome: enum[passing, failing_fixed, failing_unresolvable]`** — Reasonable. `failing_fixed` is a self-loop back to the CI gate check. But the real CI monitoring loop says "If stuck on CI failures for more than 2-3 iterations, ask the user for guidance." The design has no state or evidence field for "stuck, needs human." `failing_unresolvable` goes to `done_blocked`, but the real skill's threshold for giving up is "2-3 iterations," not a single failure. An agent following the template might submit `failing_unresolvable` after the first failure, routing to a terminal blocked state instead of continuing to try.

## 4. Free-form Mode Gap: The design's task_validation doesn't match just-do-it

The design adds `task_validation` to assess whether a free-form task is ready for direct implementation, with `verdict: enum[ready, needs_design, needs_breakdown, ambiguous]`. It explicitly treats this as analogous to "what just-do-it does."

**What just-do-it actually does is different in kind.** The real `/just-do-it` skill runs *two* jury phases — Phase 1 (Initial Jury) and Phase 3 (Validation Jury) — with a Research phase between them. Each jury phase involves multiple sub-agents (from `../issue/references/phases/phase-1-initial-jury.md` and `phase-3-validation-jury.md`). The jury output is multi-agent consensus, not a single-agent verdict. The design's `task_validation` is a single state where the agent submits `verdict: enum[...]` after some unspecified assessment. There's no mention of sub-agents, jury mechanics, or the two-phase structure.

**The design skips the research step in just-do-it.** `/just-do-it` has Phase 2 (Research) between the two jury phases. The design adds a `research` state after `task_validation`, but positions it *after* validation rather than between two validation checks. In just-do-it, research *informs* the second jury's verdict. In the design, research happens after validation is already decided. The sequence is wrong.

**The design adds a `validation_exit` terminal state; just-do-it doesn't have one.** When just-do-it determines a task is too large, it says "recommend using `/issue` + `/work-on` instead" and stops. This is a stop condition, not a workflow state. The design models it as a terminal state with a directive. That's fine for koto, but the design doesn't specify what the `validation_exit` state's directive *says* to the agent — it just says "Directive instructs the agent to communicate the verdict and suggest the appropriate next step." That's a directive *description*, not a directive.

**The scope check threshold isn't encoded.** `/just-do-it` defines explicit scope criteria: "Signs a task is too large: multiple files need significant changes, design decisions are needed, would take more than 15-30 minutes." The design's `task_validation` state doesn't carry these criteria in any evidence field or directive content. An agent submitting `verdict: needs_breakdown` does so based on what? The design doesn't say. The real just-do-it skill has those criteria spelled out. Without them in the template directive, agents will apply inconsistent thresholds.

**The `research` state in free-form mode is underdefined.** Its evidence: `context_gathered: enum[sufficient, insufficient]`, `context_summary: string`. `insufficient` has no routing in the diagram — what happens if context is insufficient? The diagram shows `research` → `setup` with no branching. If context is insufficient, does the agent keep researching? Stop? The design doesn't say, and the evidence enum implies a binary that doesn't affect routing.

## 5. Agent UX: The koto loop adds friction without clear payoff at several states

**The setup state's `mode` re-submission is confusing.** The design requires agents to re-submit `mode: enum[issue-backed, free-form]` in the `setup` evidence because koto's evidence is epoch-scoped. This is documented as a design decision, but from an agent's perspective it's surprising. The agent submitted `mode` at `entry`, moved through mode-specific states, arrived at `setup` — and now koto is asking for `mode` again. The AGENTS.md doesn't explain epoch scoping. An agent that omits `mode` from setup evidence will get an `invalid_submission` error, have to read the `details` array, figure out that `mode` is required *again*, and resubmit. This will happen on every run of this workflow. The design calls it "a straightforward authoring rule once understood" — but agents don't have context for why it's required, and the koto error message won't say "mode is required because evidence is epoch-scoped."

**The GateBlocked response shape is broken for the fallback case, right now.** The design proposes that when a gate fails on a state with an `accepts` block, koto returns a response with `expects` populated. Looking at the actual `next_types.rs`, `NextResponse::GateBlocked` has `blocking_conditions: Vec<BlockingCondition>` but no `expects` field. The design correctly identifies this as needing an engine change, but the proposed fix — "populate the `expects` field" — isn't in the current `GateBlocked` struct. An implementer would need to either add an `expects: Option<ExpectsSchema>` field to `GateBlocked`, or route through `EvidenceRequired` instead. The design says to fall through to `NeedsEvidence` in the engine, which would use `EvidenceRequired` response shape. But then the agent can't distinguish "gate failed, here's what to do" from "gate passed, but evidence is required." This matters for error messaging and for the agent understanding what happened.

**The `ci_monitor` gate command is fragile.** The command is:
```
gh pr checks $(gh pr list --head $(git rev-parse --abbrev-ref HEAD) --json number --jq '.[0].number') --json state --jq '[.[] | select(.state != "SUCCESS")] | length == 0' | grep -q true
```
This is a 4-level nested command shell-expanded at gate evaluation time. The inner `gh pr list` returns an array; if there are zero PRs (e.g., the PR was merged) or multiple PRs from that branch, `.[0].number` is null or wrong. The design notes "there is a brief window after `pr_creation` where the PR may not yet be indexed" — but that's a minimum-3-second delay in practice, during which the gate returns an error exit code, not a clean failure. The template engine runs gates synchronously; a transient API error on this gate will route the agent through the evidence fallback immediately rather than retrying. The agent has no way to distinguish "CI hasn't started yet" from "CI is failing" from "the gh command timed out." All three produce the same `GateBlocked` response.

**The self-loop states have no progress signal.** `analysis` has a self-loop on `scope_changed` and `implementation` has a self-loop on `partial_tests_failing`. From the agent's perspective, after a self-loop the agent calls `koto next`, gets the same state back with the same directive, and needs to figure out on its own what to do differently. The design says the `approach_summary` rationale field "is stored permanently in the event log" — but the agent can't read the event log to see previous attempts. koto's `koto query` command returns the current state, but the evidence from previous epochs is cleared. An agent on its third iteration through `analysis` due to `scope_changed` has no way to know it's been here twice before unless it reads the state file directly.

**The `IMPLEMENTATION_CONTEXT.md` flow is broken in the template design.** The real /work-on skill stores context injection output in `wip/IMPLEMENTATION_CONTEXT.md` and then references it in Phase 4's instructions: "Before starting implementation, review `wip/IMPLEMENTATION_CONTEXT.md` if it exists." The design's template doesn't include this artifact. The `context_injection` state's gate only checks that the issue is accessible — it doesn't create a context file. The `implementation` state's directive would need to reference the issue context some other way, or the design needs a `context_injection` state that does the actual context extraction work. As written, the design loses the design-context-carries-forward mechanism that Phase 0 and Phase 4 of the real skill depend on.

---

## Summary

The design is well-structured and the gate-with-evidence-fallback model is sound. The engine changes are correctly identified and scoped. But the document is a schema specification, not an implementation blueprint. It describes what evidence looks like but not what agents should do, it maps state names to skill phases but not state contents to phase procedures, and it has multiple state-routing gaps (introspection `Re-plan`, `finalization` `issues_found`, free-form `research` insufficiency) that would surface as stuck workflows during implementation. Any engineer picking this up for Phase 2 would need to re-read all seven skill phase files and make the same decisions the design was supposed to make for them.
