# Security Review: koto-template-authoring-skill

**Reviewer:** Security analysis agent
**Date:** 2026-03-29
**Design:** koto-template-authoring-skill (3 decision reports + summary)

## Dimension 1: External Artifact Handling

**Applicability:** Partially applicable.

The skill does not download artifacts from the network. All inputs are local:

- **New mode:** Agent-provided intent descriptions (free text typed by the user or generated by the agent).
- **Convert mode:** Reads an existing SKILL.md file from the local filesystem.

**Risks identified:**

- **Path traversal in convert mode.** The skill reads an existing SKILL.md as input. If the path to the source SKILL.md is user-supplied or agent-interpolated, a crafted path could read files outside the intended skill directory (e.g., `../../.env`). However, the agent is the one selecting the file -- there is no programmatic path resolution that an external attacker could influence. The risk is limited to the agent misidentifying which file to read, which is an agent correctness issue, not a security boundary violation.

- **Template injection via existing SKILL.md content.** In convert mode, the skill reads an existing SKILL.md and uses its content to derive phases, states, and directives. If the existing SKILL.md contains adversarial content (prompt injection in markdown), the agent processing it could be influenced. This is a general LLM prompt injection risk, not specific to this design. The compiler validation step (`koto template compile`) provides a structural check on the output, which limits the damage -- injected content would need to produce a syntactically valid koto template to survive compilation.

**Severity:** Low. No external inputs; local file reads are standard agent behavior.

**Mitigations:** None required beyond existing agent sandboxing.

## Dimension 2: Permission Scope

**Applicability:** Applicable, but low risk.

**Filesystem access:**

- Reads: existing SKILL.md files (convert mode), reference material in the skill's own `references/` directory.
- Writes: new SKILL.md file, new koto template file (in `koto-templates/` subdirectory), koto session state (managed by the koto engine in `.koto/`).
- All writes target the working directory or `.koto/` session directory. No writes to system paths, home directory dotfiles, or locations outside the project.

**Process execution:**

- Runs `koto template compile` to validate drafted templates. This is a local Go binary that parses YAML frontmatter and validates structural constraints. It does not execute the template's content -- it only checks syntax and schema.
- Runs `koto init`, `koto next`, `koto transition` for its own workflow orchestration. These are read/write operations on local JSON state files.

**Network access:** None. No HTTP calls, no API requests, no telemetry from the skill itself.

**Escalation risks:** None. No sudo, no privilege changes, no system service interaction.

**Severity:** Negligible. The permission footprint is minimal: local file I/O and execution of a trusted local binary (`koto`).

## Dimension 3: Supply Chain and Dependency Trust

**Applicability:** Minimally applicable.

The skill's dependencies are:

1. **koto binary** -- must be installed and on PATH. This is the same binary that orchestrates all koto-backed skills. Trust is established at installation time, not at skill execution time. The authoring skill does not install, update, or modify koto.

2. **Reference material** (condensed format guide, graded example templates) -- bundled in the skill's `references/` directory, shipped as part of the koto-skills plugin. These are static markdown files checked into the repository. No runtime fetching.

3. **The skill's own koto template** -- also bundled and checked in. The self-hosting aspect means the skill depends on its own template, but this is a build-time artifact, not a runtime download.

**Risks identified:**

- **Compromised koto binary.** If the koto binary on PATH is replaced with a malicious version, `koto template compile` could execute arbitrary code. This is a general supply chain risk for any CLI tool and is not introduced or amplified by this design. Mitigation is at the installation layer (tsuku's recipe verification, binary checksums).

**Severity:** Negligible for this design. Supply chain trust is handled at the koto installation layer, which is out of scope.

## Dimension 4: Data Exposure

**Applicability:** Minimally applicable.

**Data accessed:**

- Existing SKILL.md content (convert mode) -- these are developer-authored skill definitions, not secrets or PII.
- Agent-provided intent descriptions -- transient, stays in the conversation context.
- Koto session state (JSON files in `.koto/`) -- contains workflow progress (current state, variable values like skill name). No credentials or sensitive data by design.

**Data transmitted:** Nothing. The skill has no network communication. All output stays on the local filesystem.

**Risks identified:**

- **Accidental inclusion of secrets in templates.** If a user's existing SKILL.md (convert mode) contains hardcoded secrets (API keys, tokens), those could be carried into the new template. This is a user error, not a design flaw. The skill doesn't encourage or facilitate secret inclusion.

**Severity:** Negligible. No data leaves the machine, no sensitive data is accessed by design.

## Summary

| Dimension | Applicable? | Severity | Action Required |
|-----------|-------------|----------|-----------------|
| External artifact handling | Partially | Low | None |
| Permission scope | Yes | Negligible | None |
| Supply chain / dependency trust | Minimally | Negligible | None |
| Data exposure | Minimally | Negligible | None |

**Overall assessment:** This design has a minimal security surface. It reads and writes local markdown files, runs a trusted local binary for validation, and has no network interaction. The most notable risk -- template injection via existing SKILL.md content in convert mode -- is a general LLM prompt injection concern that applies to any agent reading untrusted text, and is partially mitigated by the compiler validation gate.

## Recommended Outcome

**Approve without conditions.** No security concerns warrant design changes or additional mitigations. The design's attack surface is limited to local file operations and execution of a project-local binary, both of which are standard agent behaviors already permitted by the Claude Code sandbox.
