# Phase 2 Research: UX Perspective

## Lead 1: Mermaid output strategy

### Findings

**How similar tools handle generated diagrams:**

- **Protobuf (protoc)**: Generated code lives alongside the `.proto` file OR in a separate output directory controlled by `--<lang>_out=<dir>`. The generated file is clearly derived -- it has a `// Code generated by protoc-gen-go. DO NOT EDIT.` header. Users commit generated code, and CI enforces freshness by re-generating and diffing.

- **Cargo doc**: Generates docs into `target/doc/`. The output directory is explicitly separate from source. Docs are never committed -- they're rebuilt on demand. crates.io rebuilds them centrally (docs.rs).

- **Terraform graph**: `terraform graph` outputs DOT format to stdout. Users pipe it to `dot -Tsvg` or other tools. No file is written by default. This is the purest "export" model -- the tool produces text, the user decides where it goes.

- **OpenAPI generators**: `openapi-generator generate` writes to a configurable output directory. Generated files are commonly committed and CI-checked for drift.

- **buf (protobuf linting/generation)**: `buf generate` writes to output directories specified in `buf.gen.yaml`. CI checks are first-class via `buf breaking` and `buf lint`. The generated output location is always configured, never implicit.

**The fourth option -- compile producing Mermaid as a side-effect:**

Looking at koto's current compile flow, `koto template compile <source>` prints the cached JSON path to stdout (line 570 of cli/mod.rs). The compile_cached function in cache.rs writes a `{hash}.json` file to `~/.cache/koto/`. A diagram could be written alongside the compiled JSON in the cache directory, but that defeats the purpose -- nobody looks in the cache directory. The diagram needs to be somewhere reviewable.

**Mental model analysis:**

The diagram is unambiguously a derived artifact. Users don't edit it. It's generated from the compiled template struct (states, transitions, gates). This puts it in the same category as protobuf generated code or OpenAPI client stubs -- derived, committed, CI-enforced.

The question is: derived from what, and placed where?

Currently koto has two file locations per template:
1. Source template: `templates/my-workflow.md` (in the user's repo)
2. Compiled JSON: `~/.cache/koto/{hash}.json` (in the cache, ephemeral)

The compiled JSON is ephemeral -- it's a cache artifact, not committed. The source template is the only committed file. This means the diagram's natural home is next to the source, not next to the compiled output.

**Option analysis through the UX lens:**

**(a) Separate `.mermaid.md` sibling file** (`templates/my-workflow.mermaid.md`):
- Pros: Clean separation, easy CI check (`diff <(koto template export ...) templates/my-workflow.mermaid.md`), GitHub renders it inline
- Cons: Two files per template feels like clutter; users might forget to regenerate; naming convention needs documenting
- Mental model: "The diagram is a build artifact that lives next to its source"

**(b) In-place fenced code block in source template**:
- Pros: Single file, always visible when reading the template
- Cons: Brittle -- koto must parse its own markdown output to find and replace the block. The source template's markdown body is semantically meaningful (it contains state directives extracted by `extract_directives()` in compile.rs). Injecting a mermaid block risks colliding with the state heading parser. This is not just aesthetically brittle, it's structurally dangerous given how the compiler works.
- Mental model: "The diagram is part of the template" -- but it isn't, it's derived

**(c) `koto template export` as a standalone command** (already in the design doc):
- Pros: Clean, composable (stdout by default), follows terraform graph pattern
- Cons: "Nobody runs it" -- the user's concern. But this is solvable with CI enforcement, which is already in scope.
- Mental model: "The diagram is produced on demand"

**(d) Compile producing diagram as side-effect**:
- Pros: Zero extra steps for the user
- Cons: Violates the existing design decision that preview is its own command. Overloads compile's contract (YAML in, JSON out). The design doc explicitly rejected flags on compile for this reason.
- The user specifically said interactive preview should NOT be a side-effect. But they didn't say the same about Mermaid text. Still, adding implicit file writes to compile changes its contract.

**The least-friction path that doesn't feel weird:**

Option (c) -- `koto template export --format mermaid` -- already exists in the design doc. The "nobody runs it" concern is addressed by the GHA workflow (also in scope). The pattern is:

1. Author runs `koto template export --format mermaid --output templates/my-workflow.mermaid.md` during development
2. CI runs the same command and diffs against the committed file
3. If they diverge, CI fails

This is exactly how buf, protoc, and OpenAPI generators work. The file lives next to the source template. The naming convention is `<stem>.mermaid.md`. CI enforces freshness.

The "separate file feels weird" concern likely comes from the unfamiliarity of having two files per template. But this is standard practice for any source-with-generated-artifact pattern. And the alternative (in-place update) is genuinely dangerous given koto's markdown parsing architecture.

### Implications for Requirements

1. **`koto template export` is the right command** -- it already exists in the design. The PRD should confirm this, not introduce a new mechanism.
2. **The diagram file lives next to the source template**, not next to compiled output (which is ephemeral cache).
3. **Naming convention**: `<template-stem>.mermaid.md` enables GitHub's native Mermaid rendering and is distinguishable from the source `.md`.
4. **In-place source update (option b) should be explicitly rejected** -- the compile.rs `extract_directives()` function treats H2 headings as state boundaries. A `## Diagram` section would either need special-case exclusion or risk being treated as a state name. This is a hard technical constraint, not just a style preference.
5. **CI enforcement is the answer to "nobody runs it"** -- the GHA workflow is the mechanism that makes export non-optional.

### Open Questions

1. Should `koto template export` support `--output` defaulting to `<stem>.mermaid.md` (like preview defaults to `<stem>.preview.html`), or should it always default to stdout? Stdout is more composable; a smart default reduces friction for the common case.
2. Should the export command also accept pre-compiled JSON, or only source templates? The design doc says both. This matters for CI workflows where compile and export might be separate steps.
3. Is `.mermaid.md` the right extension? Alternatives: `.diagram.md`, `.graph.md`, just `.mermaid`. GitHub renders fenced mermaid blocks inside any `.md` file, so the extension is for human convention, not tool support.


## Lead 2: User journeys

### Findings

**Journey 1: Template Author (create -> compile -> debug -> iterate -> commit -> PR)**

Current workflow based on code analysis:

```
1. Write template source (.md with YAML front-matter)
2. Run: koto template compile templates/my-workflow.md
   -> Prints cache path to stdout (e.g., ~/.cache/koto/abc123.json)
   -> On error: prints JSON error to stdout, exits non-zero
3. Inspect compiled output: cat ~/.cache/koto/abc123.json | jq .
4. Test: koto init my-test --template templates/my-workflow.md
5. Walk through: koto next my-test [--with-data '...'] [--to state]
6. If stuck: read the JSON, try to trace transitions manually
7. Iterate: edit template, go to step 2
8. Commit template source
9. Open PR
```

Where visual tooling fits:

- **Step 3 (inspect compiled output)**: This is where `koto template export --format mermaid` adds value. Instead of reading JSON to understand graph shape, the author sees a diagram. This is a natural checkpoint -- "does the graph look right before I test it?"
- **Step 6 (debugging stuck workflows)**: This is where `koto template preview` adds value. The interactive HTML with click-to-highlight lets the author trace paths through the graph. Tooltips show gate conditions and evidence schemas without reading JSON.
- **Step 8-9 (commit and PR)**: The committed `.mermaid.md` file gives the author (and reviewers) a visual diff of what changed.

Natural vs interruption:
- Export after compile (step 3): Natural. It's the "did I get the structure right?" checkpoint.
- Preview during debugging (step 6): Natural. It replaces "stare at JSON" with "click through the graph."
- Export before commit (step 8): Slight friction -- it's an extra step. CI enforcement makes it mandatory rather than optional, which is better than hoping authors remember.

**Journey 2: PR Reviewer (see PR -> understand changes -> approve/request changes)**

Current workflow:

```
1. Open PR on GitHub
2. See diff of templates/my-workflow.md (YAML front-matter changes)
3. Try to understand what the structural change means
4. Read through YAML manually to trace state graph changes
5. Approve or request changes
```

Where visual tooling fits:

- **Step 3-4**: If a `.mermaid.md` file is committed and changed in the PR, GitHub renders the Mermaid diagram inline in the diff view. The reviewer sees the before/after graph structure without parsing YAML. This is the highest-value insertion point for Mermaid.
- The reviewer does NOT run `koto template preview` -- they're on GitHub, not in a terminal. The committed Mermaid diagram is their only visual aid.

Natural vs interruption:
- Seeing the Mermaid diff in the PR: Completely natural. It's just another file in the diff, but GitHub renders it visually.
- Running preview locally: Interruption. Reviewers shouldn't need to check out the branch and run commands to understand a template change.

This reinforces that the committed Mermaid file is the critical artifact for reviewers, not the interactive preview.

**Journey 3: Repo Maintainer (set up CI -> configure workflow -> enforce standards)**

Current workflow:

```
1. Templates exist in the repo
2. No CI validation beyond "does it compile"
3. Diagrams (if any) can drift from source with no enforcement
```

With visual tooling:

```
1. Add reusable GHA workflow to .github/workflows/
2. Configure: list of template source paths (or glob pattern)
3. On PR:
   a. Compile each template (validates syntax)
   b. Export Mermaid for each template
   c. Diff exported Mermaid against committed .mermaid.md
   d. Fail if they diverge
4. Maintainer reviews CI output, approves/blocks PRs
```

Where visual tooling fits:

- **Step 2-3**: The GHA workflow is the enforcement mechanism. The maintainer configures it once.
- **Step 3d**: The "freshness check" is the key CI innovation. Without it, committed diagrams become stale documentation.

Natural vs interruption:
- Setting up the GHA workflow: One-time configuration, natural for any repo maintainer.
- CI blocking on stale diagrams: Natural for repos that use CI. The error message should clearly say "run `koto template export --format mermaid --output <path>` to update."

**Cross-journey observations:**

1. The template author and PR reviewer have complementary needs: the author generates the diagram, the reviewer consumes it. The Mermaid file is the bridge.
2. The interactive preview serves only the template author (local debugging). It has no role in the reviewer or maintainer journey.
3. CI enforcement serves the maintainer journey but also helps the author journey (catch forgotten regeneration before the PR is reviewed).
4. The compile step is already the natural checkpoint in the author journey. Adding export as an explicit next step (not a side-effect) fits the existing flow.

### Implications for Requirements

1. **The Mermaid diagram is the reviewer's primary artifact** -- it must be committed and CI-enforced, not optional.
2. **The interactive preview is author-only tooling** -- it doesn't need to be committed, shared, or CI-integrated.
3. **The GHA workflow needs clear, actionable error messages** -- when the Mermaid file is stale, the error should include the exact command to fix it.
4. **Template export should be easy to integrate into a pre-commit hook** as an alternative to CI-only enforcement. Some authors will want local enforcement too.
5. **The reviewer journey depends entirely on GitHub's Mermaid rendering** -- the PRD should note this as a platform dependency and verify that `stateDiagram-v2` renders correctly in GitHub PR diffs.

### Open Questions

1. Should the GHA workflow post a PR comment with the rendered diagram, or is the diff view of the `.mermaid.md` file sufficient? PR comments add visibility but also noise.
2. For repos with many templates (10+), should the GHA workflow run in parallel per template or sequentially? This affects CI time.
3. Should the PRD specify a pre-commit hook option alongside CI? Local enforcement catches issues before push.


## Lead 3: Preview command interaction

### Findings

**Compile step weight:**

Reading `compile.rs`, the compilation process is:
1. Read the source file (`std::fs::read_to_string`) -- I/O bound, fast for small files
2. Split front-matter (`split_frontmatter`) -- string scanning, trivial
3. Parse YAML (`serde_yml::from_str`) -- deserialization, fast for typical template sizes
4. Extract directives (`extract_directives`) -- line-by-line scan, trivial
5. Build compiled states -- iteration over HashMaps, trivial
6. Validate transition targets -- map lookups, trivial
7. Validate the template (`template.validate()`) -- structural checks

There are no network calls, no heavy computation, no file system traversal. The compile step is purely CPU-bound string processing and deserialization. For a 30-state template (the upper bound mentioned in the design doc), this would take single-digit milliseconds at most.

The cache layer (`cache.rs`) adds SHA256 hashing and a file write on cache miss, plus a file existence check on cache hit. Still sub-millisecond for the hash, and a fast filesystem operation for the write.

**Preview generation weight:**

Based on the design doc, `generate_preview()` does:
1. Serialize the `CompiledTemplate` to JSON (`serde_json::to_string`) -- fast
2. String replacement of the placeholder in the embedded HTML -- one `str::replace` call
3. Write the HTML file -- single file write

Total: well under 100ms for any realistic template size.

**Browser opening weight:**

The `opener` crate invokes the OS default handler. On Linux, this is `xdg-open`. The browser may take 1-2 seconds to open or switch tabs, but the koto command returns immediately after launching the process.

**Is re-running sufficient vs watch mode?**

The complete cycle (edit template -> run `koto template preview` -> see result in browser) takes:
- Edit: user time
- Run command: <100ms for compile + preview generation
- Browser refresh: 1-2s (browser tab switching/loading)

Total tool overhead: under 2 seconds. This is comparable to `cargo doc --open` which also has no watch mode and nobody complains.

Watch mode would save the user from switching to the terminal and typing the command. But it adds:
- A file watcher dependency (notify crate or similar)
- A local server OR file-system polling in the browser
- Process lifecycle management (Ctrl+C to stop)
- Complexity in error handling (what if the template is in an invalid state mid-edit?)

The design doc already decided "no local server for v1." Watch mode without a server means regenerating the HTML file and relying on the user to manually refresh the browser -- which saves only the "type the command" step, not the "refresh browser" step.

**Verdict**: Re-running is sufficient for v1. The compile and preview generation steps are fast enough that manual re-run adds negligible friction. Watch mode is a nice-to-have for v2 if users request it, but it's not needed to make the debugging workflow functional.

### Implications for Requirements

1. **No watch mode in v1** -- the compile step is fast enough that re-run is acceptable.
2. **The preview command should print a clear message** like "Preview written to my-workflow.preview.html" so the user knows which file to refresh in their browser.
3. **If watch mode is added later**, it should include a local server with auto-reload (otherwise the user still manually refreshes, defeating the purpose). That's a v2 feature.
4. **The PRD should document the expected latency** (<100ms for compile + generate) so implementers know not to add unnecessary optimization.

### Open Questions

1. Should the preview command automatically refresh an already-open browser tab? Some tools do this via AppleScript/xdotool, but it's platform-specific and fragile. Probably not worth it.


## Summary

The Mermaid diagram should live as a sibling file next to the source template (e.g., `<stem>.mermaid.md`), produced by the already-designed `koto template export` command and enforced by CI freshness checks -- this matches how protobuf, OpenAPI, and buf handle generated artifacts. In-place source template updates should be explicitly rejected because koto's compiler parses H2 headings as state boundaries, making injected content structurally risky. Watch mode for the preview command isn't needed in v1: compilation takes single-digit milliseconds and preview generation under 100ms, making manual re-run a sub-2-second cycle that matches `cargo doc --open` ergonomics.
