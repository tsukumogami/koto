# Decision 6: --check implementation

## Alternatives

### a. In-memory byte comparison

Generate output into a `Vec<u8>`. Read the existing file into a second `Vec<u8>`. Compare with `==`.

- **Memory**: Two buffers. For realistic template exports (Mermaid text ~2-5 KB, HTML ~30-80 KB), this is negligible.
- **Error messages**: Can report that files differ, but cannot show a meaningful diff without additional work. A "files differ" message plus the fix command is sufficient for CI -- developers run the fix command, not read diffs of generated HTML.
- **Simplicity**: Straightforward. No external dependencies, no temp files, no hashing. The generation path already produces bytes (or a String that converts trivially).
- **Works for both formats**: Yes, both mermaid and html are text buffers.

### b. Tempfile + diff

Write generated output to a `NamedTempFile`. Shell out to `diff` (or similar) to compare against the target path.

- **Memory**: Same -- the generated content must exist in memory before writing to the tempfile anyway.
- **Error messages**: Could show a unified diff, but for generated HTML this is noisy and unhelpful. Mermaid diffs might be readable, but the primary audience is CI, not humans reading diffs.
- **Simplicity**: Requires `tempfile` crate, spawning a subprocess, handling `diff` not being available on all platforms (Windows), and parsing exit codes from the child process.
- **Works for both formats**: Yes, but overkill.

### c. Streaming hash comparison

SHA256 the generated bytes and the existing file bytes, compare digests.

- **Memory**: Could stream the file read through a hasher to avoid holding the full file in memory. But we already hold the generated content in memory, so we'd only save one buffer. Not meaningful at these sizes.
- **Error messages**: Can only report match/mismatch, same as (a). Strictly less capable since you can't even compare lengths or show a prefix of the difference.
- **Simplicity**: Requires a SHA256 dependency (the crate already uses `sha2` for cache hashing, so it's available). More code than a direct comparison, for no benefit.
- **Works for both formats**: Yes.

## Recommendation

**Option (a): In-memory byte comparison.**

It's the simplest approach, has zero additional dependencies, and matches the scale of the problem. The generated content is already a `String` or `Vec<u8>` in memory. Reading a second copy of a file under 100 KB costs nothing. Hashing adds complexity without benefit -- you still can't show a diff, and you don't save meaningful memory. The tempfile approach introduces platform-dependent behavior and an external process for no gain.

The comparison should be exact byte equality, matching the PRD's requirement for deterministic output with LF line endings (R5). No normalization, no fuzzy matching.

## Key interfaces (Rust types/signatures)

```rust
/// Check whether generated content matches the existing file at `path`.
/// Returns Ok(true) if identical, Ok(false) if they differ or the file
/// doesn't exist. Returns Err only for I/O errors other than NotFound.
fn check_freshness(generated: &[u8], path: &Path) -> io::Result<CheckResult> {
    match std::fs::read(path) {
        Ok(existing) => {
            if generated == existing.as_slice() {
                Ok(CheckResult::Fresh)
            } else {
                Ok(CheckResult::Stale)
            }
        }
        Err(e) if e.kind() == io::ErrorKind::NotFound => Ok(CheckResult::Missing),
        Err(e) => Err(e),
    }
}

enum CheckResult {
    Fresh,   // File exists and matches generated output
    Stale,   // File exists but content differs
    Missing, // File does not exist
}
```

The caller in the CLI `run()` function handles each variant:

```rust
match check_freshness(generated.as_bytes(), output_path)? {
    CheckResult::Fresh => {
        // exit 0, optionally print confirmation to stderr
        std::process::exit(0);
    }
    CheckResult::Stale => {
        eprintln!("error: {} is out of date", output_path.display());
        eprintln!("run: koto template export --format {} --template {} --output {}",
                   format, template_path, output_path.display());
        std::process::exit(1);
    }
    CheckResult::Missing => {
        eprintln!("error: {} does not exist", output_path.display());
        eprintln!("run: koto template export --format {} --template {} --output {}",
                   format, template_path, output_path.display());
        std::process::exit(1);
    }
}
```

## Error message format

Plain stderr, not JSON. This aligns with the design doc's note that `export` and `preview` are developer-facing commands, not agent-consumed. The existing JSON error pattern (`exit_with_error`) is used by `init`, `next`, `transition`, and `rewind` -- commands in the agent-facing workflow loop. Template export is a build/CI tool.

Format:

```
error: docs/workflow.html is out of date
run: koto template export --format html --template workflow.md --output docs/workflow.html
```

For missing files:

```
error: docs/workflow.html does not exist
run: koto template export --format html --template workflow.md --output docs/workflow.html
```

Exit code 1 for both stale and missing. There's no value in distinguishing these with different exit codes -- CI just needs pass/fail, and the stderr message already explains the situation.

## Confidence: High

The file sizes are small, the comparison is trivial, and the approach has no dependencies beyond `std::fs::read`. The only design choice with any nuance is JSON vs plain stderr, and the design doc already made that call.
